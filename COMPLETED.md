### 13. [x] "New this week" badges — 5ab4a01 (design note + ROADMAP flip) / ac4ae3b (helpers + render + tests) / 4cbfc68 (workflow)

Ellen scans the republished page looking for events she hasn't already internalised. With a 60-day lookback and ~30–60 rendered cards, fresh extractions blended into the wall of already-known events. Per-run delta badge, rendered inline with each card's `event-name` div, persisted across runs via a state-branch manifest. Full design at `design/new-this-week-badges.md`.

Persistence lives in its own file, `prior_events.json`, not inside `events_state.json`. Separate concerns: `events_state.json` is the Gmail-message extraction cache with a load-bearing schema and parity contract with `events_state.py`; `prior_events.json` is a render-time artefact ("what was on the published site last run"). File-per-concern mirrors the existing state-branch pattern alongside `.filter_audit.json` / `blocklist_auto.txt` / `events_state.json`. Format is minimal — `{"generated_at_iso": ..., "event_ids": [sorted]}`; nothing downstream parses the timestamp.

Missing-vs-empty semantics are load-bearing: `_load_prior_event_ids` returns `None` when the file is missing / unreadable / malformed / wrong-shape, and returns `set()` (possibly empty) when the file is present and well-formed. The caller writes `new_ids = (current_ids - prior) if prior is not None else set()`, so first-run (no manifest) suppresses every badge — flashing NEW on every card on first publish would be visually useless. A legitimate empty-prior state (rare: last run rendered zero events) still badges everything as new. Four loader edge-case tests and a round-trip saver test pin the contract. Saver uses the same `.tmp` + `os.replace` atomic-write pattern as `events_state.py::save_state`.

Render wiring threads `new_ids: set[str] | None` through `render_html` and both card functions (`_event_card`, `_undated_card`). Badge HTML is `<span class="new-badge">NEW</span>` injected inside the `event-name` div before its closing tag — content signal, not action; tight visual association with the title. CSS rule is one accent-coloured inline pill added to the `<style>` block; the rule always ships regardless of whether any card renders the span, so tests assert on the rendered `<span class="new-badge">` element rather than the `.new-badge` selector. Undated cards get the badge via the same code path (both stamp `data-event-id` and both own an `event-name` div). Ignored-but-new cards render the badge in HTML — `display:none` from the `.ignored` rule hides it until Show ignored is clicked, at which point the NEW badge correctly signals a newly-extracted auto-ignored card. `_render_basic_mixed(new_ids)` fixture helper keeps the render-behavior tests terse.

Design note locked the scope as render-only — no changes to `events_state.py`, the extractor prompt, or the event-ID hashing scheme. One minor deviation from the design note's "Files touched" list: `main.py` was edited to wire `--prior-events PRIOR_EVENTS_PATH` through `step4_process_events` (design note had said "No changes to main.py"). Deliberate — threading the path from the pipeline entrypoint keeps the persistence boundary symmetric with the existing state-branch-file pattern; the alternative (hard-coded path inside `process_events.py`) would have obscured the coupling to the workflow's restore/save steps.

Workflow plumbing (`.github/workflows/weekly-schedule.yml`): restore block copies `.state/prior_events.json` → repo root when present with an event-count log line ("no prior — NEW badges suppressed this run" on first-ever run); save-block early-exit guard includes `prior_events.json` so a run producing only the manifest still commits; save-block `FILES` list adds `prior_events.json` alongside `events_state.json`. Save step's existing `if: dry_run != 'true'` gate means dry runs render badges against the last real prior-run state but do not advance the manifest — next real run's diff stays correct.

"New" is binary per run — no aging (no "stays NEW for 2 cadence cycles"). Each run's render overwrites the manifest with the current union, so a Mon-new event loses its badge by Wed. Simplest possible semantic and matches the ROADMAP wording.

ID source is the event dict's `id` field (already stamped by `events_state.stamp_event_ids` upstream of render). No recomputation at badge time — avoids drift between the cache's ID view and the render's ID view, and keeps the distinction worthwhile because `prior_events.json` and `events_state.json` GC on independent schedules: cache eviction does not touch the render manifest, and a cache-cleared run still diffs correctly against prior because event IDs are deterministic from `(name, date, child)`.

Test coverage: 17 new tests in `tests/test_process_events.py` under the `# ─── prior-events manifest + NEW badges (#13) ─` section — 7 loader edge cases (missing file, empty path, empty list, happy path, malformed JSON, wrong-shape JSON list, missing `event_ids` key), 3 saver tests (round-trip, sorted-and-stamped payload, atomic `.tmp` + rename), 7 render behaviours (dated card with badge, dated card without badge, undated card with badge, first-run `new_ids=None` suppression, `new_ids=set()` empty-set suppression, CSS rule always present, badge survives on an ignored card via the `ignored_and_sender.json` fixture).

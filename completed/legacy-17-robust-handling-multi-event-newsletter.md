# 17. Robust handling of multi-event newsletter emails — 2f68501 (design + ROADMAP flip) / 85ae9fa (stats module) / 89fe4be (--reextract) / bcee931 (agent batching) / 191edaf (main stats integration) / 00d0a19 (digest bridge) / 3d4bcaa (workflow + .gitignore)

Newsletters (LAES PTA Sunbeam, FCPS updates, similar school-list digests) routinely carry 5–15+ calendar items in a single email. The extractor was already prompted to pull every item (see `agent.py` rule 8, "Newsletter calendar items"), and per-batch parsing treated N events from one `source_message_id` as the normal case. What was missing: any *signal* when the extractor under-extracted one of these issues, and any *affordance* to re-process a message after its ID was recorded in `events_state.json::processed_messages` (the incremental-extraction cache would otherwise skip it forever). Scope covered four inter-locking pieces shipped as seven self-contained commits. Full design at `design/newsletter-robustness.md`.

**The stats module (`newsletter_stats.py`, 85ae9fa).** A file-per-concern companion to `events_state.json` and `prior_events.json`: `sender_stats.json` records per-sender lifetime counters (`messages_seen`, `total_events`), a 10-entry rolling window of per-message event counts (FIFO; newer appends, older fall off the front), first/last-seen timestamps, and a sticky `is_newsletter` flag. Public API is pure plus two I/O endpoints (`load_stats` / `save_stats`); everything else — `update_sender_counts`, `classify_senders`, `newsletter_senders`, `outlier_alerts` — mutates in-memory dicts. Atomic writes via `tempfile` + `os.replace` mirror `events_state.save_state`. Tolerant load mirrors it too: missing / malformed / wrong-schema / unreadable all warn-and-fall-back to an empty stats object. Classification is sticky (`False → True`, never the other direction — manual file edit is the escape hatch). 31 unit tests pin every invariant: load/save round-trip, rolling-window size bound, zero-count messages contributing `0` to the window, promotion thresholds at `messages_seen >= 3 AND median(per_message_counts) >= 5`, sticky promotion, outlier threshold at `max(2, round(prior_median * 0.5))` with strict `<` for the flag, floor of 2 protecting low-prior-median senders from false-positive alerts.

**The `--reextract` CLI (`main.py`, 89fe4be).** `python main.py --reextract <MESSAGE_ID>` evicts *both* `processed_messages[mid]` and every event whose `source_message_id == mid` from `events_state.json`, saves the evicted state to disk so step2c sees it, and runs the pipeline normally thereafter — step2 refetches the message, step3 sends it to the extractor. Independent utility: Tom can use it from a manual `workflow_dispatch` the moment C3 lands, no later commit required. Unknown message IDs warn-but-do-not-fail (fat-finger safety). Purging source-matched events is destructive by design — the re-extraction merges its own events back in, but any originals the re-extraction doesn't reproduce stay gone. That's the whole point of re-extracting.

**Newsletter-isolated agent batching (`agent.py`, bcee931).** Two pure helpers: `_sender_key(from_header)` parses a `From` header via `email.utils.parseaddr` and returns the lowercased mailbox — not the domain. Domain-level aggregation would dilute the signal across high-yield and low-yield senders on the same domain (e.g. one FCPS list yields 10 events an issue, another FCPS list yields 1). `_plan_batches(emails, newsletter_senders)` partitions inputs into newsletter emails (each in its own batch-of-1 for maximum per-email agent attention) and regular emails (`BATCH_SIZE=10` chunks as before). Newsletter batches run *first* so a cheap regular-batch parse failure doesn't gate expensive newsletter work. `extract_events` grew `newsletter_senders: set[str] | None = None`; `None` preserves pre-#17 batching bit-for-bit and was load-bearing for C4 landing independently of C5. 16 tests pin the pure-helper behaviour, including the regression property (`None` → identical chunking), case-insensitive match, named-form mailbox extraction, mixed-partition ordering, and a signature guardrail. Edge case: garbage `From` headers (e.g. `"not an email"`) return a bare token via parseaddr, not empty-string — the safety invariant is "never matches a real mailbox", property-tested rather than asserting a specific string.

**Pipeline integration (`main.py`, 191edaf).** Stats loaded *before* extraction so `ns.newsletter_senders(stats)` can drive batching. Outlier alerts computed *before* `update_sender_counts` so `prior_median` reflects history only — folding the current run in first would bias the threshold toward a short current count and mask the alert it's meant to fire. `_per_message_counts(new_emails, candidates)` seeds triples from `new_emails` (not from candidates), which is what lets a quiet-newsletter-issue that yielded zero events still contribute `0` to the rolling window and keep the median honest rather than drifting upward. Dry-run reads stats but does not save them (mirrors #13's `prior_events.json` handling). Stats save is double-gated: `not --dry-run AND per_message_counts non-empty` — an all-cached run produces no new triples and leaves the file untouched. The C5 alerts surface was intentionally stdout-only ("STEP 3c" banner with the message ID verbatim for copy-paste into `--reextract`); the digest surface landed in C6.

**Digest bridge (`scripts/process_events.py` + `main.py`, 00d0a19).** `_load_outlier_alerts` is the tolerant list loader (missing/malformed → `[]`). `_render_outlier_block_text` and `_render_outlier_block_html` are pure renderers returning `list[str]` so the digest builders can `extend()` unconditionally. HTML block escapes `sender` and `message_id` since both flow from untrusted Gmail headers; XSS pinned by a test that injects `<script>` into `sender` and asserts on `&lt;script&gt;`. `render_digest_text` and `render_digest_html` grew `alerts=None` kwargs and inject the ⚠️ Possible under-extraction block between the subject and the event list — primes Tom to check whether anything important is missing *before* he scans the events. `main.py::step4_process_events` grew `outlier_alerts` kwarg: writes a `NamedTemporaryFile` `-alerts.json` when non-empty, appends `--outlier-alerts <path>` to `script_args`, extends the existing cleanup `finally` block. Empty/None short-circuits — no tempfile, no flag, no surface. Monday-only digest surfacing is implicit via step6's existing cadence gate; Wed/Sat runs still write the alerts tempfile, but the digest draft never ships to Gmail, so Wed/Sat alerts land in the Actions log only.

**Workflow plumbing (`.github/workflows/weekly-schedule.yml` + `.gitignore`, 3d4bcaa).** Three surgical additions mirroring the `prior_events.json` plumbing from `4cbfc68` (#13): restore step reads `.state/sender_stats.json` with sender-count reporting and first-run fallback; save step's early-exit AND-guard, `cp ../...` copy block, and conditional `FILES=""` list each gain the file. `.gitignore` adds `sender_stats.json` under the Bot-generated artifacts block with an in-file comment pointing at the state branch. First-run-after-deploy note: on the very next workflow run, `.state/sender_stats.json` is absent so the classifier boots empty; no sender can be promoted on that run because promotion requires `messages_seen >= 3` over the rolling window. Two more weekly cadences then bring the first eligible newsletter across the threshold. Warm-up is intentional.

**Key design decisions worth preserving.** Sender key is lowercased mailbox, not domain — single-source the canonicalization via `agent._sender_key` imported into `main.py` so fixing one place fixes both. Promotion thresholds are `messages_seen >= 3 AND median >= 5`; demotion is not implemented (manual JSON edit only). Outlier threshold is `max(2, round(prior_median * 0.5))` with *strict* `<` — `<=` would fire on any sender whose current count exactly matched half its prior, which is not outlier enough to be worth interrupting Monday reading. Rolling window capped at 10 for bounded file growth and to keep the median insensitive to year-old data. Zero-yield runs count so a quiet summer newsletter issue contributes `0` to the rolling median rather than inflating it. Newsletter-isolated batches run first so cheap-batch failure can't gate expensive work. Alerts surface in the Monday digest AND the Actions log but NOT on the published site — the schedule page is a live view, not a status dashboard. Dry-run reads stats but does not save them, same as #13's prior_events handling.

**One ordering invariant that cuts across commits.** `outlier_alerts(stats, per_message_counts)` must be called BEFORE `update_sender_counts(stats, per_message_counts, now_iso)`. The docstring on the stats module says it; `main.py` step3c does it; test `test_outlier_alerts_uses_prior_stats_not_current` pins it. If a future refactor inlines the two calls, or swaps their order "to be tidy", the alert threshold silently starts reflecting the current run's (potentially short) count and the feature stops detecting under-extraction. Load-bearing.

# 12. Per-kid filter chips — f0976f6 (design note) / fd0c264 (roster subtask) / 399d383 (chips)

A filter row above the event list so Tom can hide events that aren't for a given kid. Landed as three commits in session 5 on top of the card redesign (#11), ROADMAP flipped `[ ]` → `[~]` in `0f1c3f5`, then flipped to `[x]` in session 6 after Tom's live-page visual QA once #19's deterministic attribution made the chips actually useful on the messier audience-string cases ("6th grade AAP", "Cuppett Performing Arts").

Design note at `design/per-kid-filter-chips.md` (`f0976f6`) locks the shape. Teacher roster subtask (`fd0c264`) added `class_roster.json` at the repo root and wired it into `agent.py` via `_load_roster_prose` (read at module import) + `_format_roster_prose` (pure, unit-tested). The loader crashes on missing/malformed roster rather than falling back silently — the file is committed, so absence is a bug. Chip row + filter JS (`399d383`) adds the three buttons, the `data-child` attribute on every event card, and the click handler that toggles a body-level class. Later enhanced in `cd816f0` with an optional `activities` key per kid (empty list or absent → no clause) so future kid additions don't require a code change.

Locked decisions worth preserving for future UI changes:

- **Chip set is hard-coded to three buttons (`All` / `Everly` / `Isla`).** It does NOT iterate over unique children in the run — guarded by `test_filter_chip_row_is_static_not_derived_from_events`. Free-text audience values ("All LAES students", "6th grade AAP") are an unstable taxonomy and were deliberately excluded. Adding a third kid requires a one-line edit to the chip list (and a matching roster row) — deliberate friction so the UI doesn't silently grow with every transient audience string the extractor produces.
- **Filter semantics are non-lossy.** Clicking `Everly` hides only `data-child="isla"` cards; audience-line cards and empty-`data-child` cards stay visible under every selection. This is why a school-wide closure still surfaces while you're filtered to Isla, and why #19's "All LAES students" behavior (data-child="") stays correct — shared-school events attribute to nobody and show under every chip. If Tom ever asks for a strict "only this kid" view, it's a two-line CSS change.
- **Filter state is ephemeral — no localStorage key.** Cold reload returns to `All`. Intentional; keeps the filter distinct from the `ignore` localStorage bookkeeping, so there's no hidden-state surprise where a URL share or cold open mysteriously hides cards.
- **The hide rule uses `display: none !important`** for specificity symmetry with the existing `.show-ignored .event-card.ignored` override. Without `!important` an Everly-filter + Show-ignored combo would leak an Isla ignored card. Covered by `test_filter_hide_css_uses_important`.
- **Roster file shape:** `class_roster.json` at repo root is the source of truth; edit-and-commit is the whole update flow. Current values (as of session 6): Everly in 6th grade (Ms. Anita Sahai) with activity "Born 2 Dance Studio (B2D)"; Isla in 3rd grade (Ms. Meredith Rohde) with activity "Cuppett Performing Arts Center"; both at Louise Archer Elementary. `agent.py` reads it at import and injects the prose into `EXTRACTION_SYSTEM_PROMPT`; `roster_match.py` (added in #19) reads it at import for the deterministic attribution path.

Test coverage: 8 chip tests (static chip row, hide CSS specificity, non-lossy semantics, etc.) + 3 roster-prose tests + 1 activities-clause test. Tom's visual QA on the live GitHub Pages build after #19 landed confirmed chips filter the expected cards while keeping audience-string and empty-`data-child` cards visible across every selection.

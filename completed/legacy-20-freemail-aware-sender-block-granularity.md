# 20. Freemail-aware sender-block granularity — f855dee (design + ROADMAP flip) / 745957a (freemail list + loader + tests) / d5820c2 (main.py derivation) / 563354c (events_state v3 bump) / bf9fe35 (process_events render wiring) / 8170081 (is_protected address-aware) / 03b44c5 (build_queries docs + integration test) / e448a8a (apps_script SENDER_RE)

Tom surfaced the problem mid-session 10 with a screenshot of the live site: two unrelated `@gmail.com` senders — a swim coach and a teacher — both rendering "Ignore sender (gmail.com)". Clicking either would have silently blocked every other `@gmail.com` sender, including family, classmates' parents, and any institutional account that uses a freemail address for replies. The original #6+7 Ignore-sender flow blocked at the registrable-domain level unconditionally via `tldextract.top_domain_under_public_suffix`, which is the right granularity for schools and team-management platforms but catastrophic for consumer-email senders. Scope: make the granularity sender-aware — one address at a time for known freemail domains, unchanged domain-level behavior for everything else — while preserving the existing protected-domains guardrail end-to-end. Full design at `design/sender-block-granularity.md`.

**Two-field data model per event.** Events carry both `sender_domain` (registrable domain — unchanged semantic, still drives `is_protected` and the Gmail-query exclusion shape for institutional senders) and new `sender_block_key` (what the Ignore-sender button POSTs and what the Ignored Senders sheet row stores: full lowercased address for freemail, equal to `sender_domain` otherwise). Main.py's `_attach_sender_domains` grew sibling `_attach_sender_block_keys` that reads `freemail_domains.txt` once at import and picks the right shape per event from its parsed `From` header. Rendering, client JS, Apps Script webhook, Gmail-query exclusion, and `is_protected` all consume the block key for action and the domain for policy. A third possible design — single field that switches shape — would have produced mixed-granularity cards during the 120-day state GC window (old rows with domain-only keys rendering next to new rows with address keys on the same sender), which is confusing UX; the two-field approach avoids that entirely and the schema-v3 blow-away retires the ambiguous-shape rows in one pass.

**Freemail list is hand-curated, not derived.** `freemail_domains.txt` at repo root seeded with 19 common consumer-email hosts (gmail.com / yahoo.com / outlook.com / hotmail.com / icloud.com / aol.com / me.com / mac.com / live.com / msn.com / comcast.net / verizon.net / protonmail.com / proton.me / fastmail.com / gmx.com / gmx.us / yandex.com / zoho.com). Loader `freemail.load_freemail_domains(path)` mirrors `protected_senders.load_protected_senders` — tolerant of comments and blank lines, case-folded on read, returns an empty list when the file is missing. Deliberately not computed from MX records or DNS — the list changes rarely, a one-line edit is cheap, and the alternative adds network dependencies to the build pipeline for minimal gain. New hosts land via Tom adding the line and committing; the sticky part is knowing when a given domain is "consumer-facing" vs. "single-org vanity domain", which requires human judgment the pipeline shouldn't try to automate.

**Address-awareness is the load-bearing protected-domains guarantee.** The critical invariant Tom pinned this session: `alice@fcps.edu` must trip the same `fcps.edu` protection as a bare `fcps.edu` entry in the Ignored Senders sheet. Without this, any address-form row in the ignored-senders cache could — in principle — land a protected institutional domain in the Gmail-query exclusion, with the only defense being the UI-level button suppression (which doesn't prevent direct sheet edits or stale pre-#20 rows). `protected_senders.is_protected` now splits on `@` when present and matches the domain part, making the check shape-agnostic. Both consumers (`process_events.render_html` suppressing the Ignore-sender button; `build_queries.main` filtering protected domains out of the ignored_senders union) share this one matcher — expanding it in one place locks the guarantee across both surfaces. The C7 integration test (`test_build_queries_drops_address_form_protected_from_ignored_senders`) pins the end-to-end path: an address-form `alice@fcps.edu` in `ignored_senders.json` drops from the Gmail clause, while an address-form `alice@gmail.com` (freemail, not protected) lands in the clause unchanged.

**Wire-protocol key stays `payload.domain`.** The Apps Script `ignore_sender` / `unignore_sender` endpoints still read `payload.domain`, but the regex gates opened up: `DOMAIN_RE = /^[a-z0-9][a-z0-9.-]*\.[a-z]{2,}$/` became `SENDER_RE = /^(?:[^\s@]+@)?[a-z0-9][a-z0-9.-]*\.[a-z]{2,}$/` — strictly broader, so first-wave clients still sending domain-only payloads continue to validate with no flag-day rename in client JS. Four call sites swapped (`_handleIgnore` on the event-row sender column, `_handleIgnoreSender` on the sender-row block identifier, `_handleUnignoreSender` as the match target for bulk delete, `_listIgnoredSenders` on the read route). Unignore-sender exact-match semantics are called out in the header comments: `alice@gmail.com` does NOT match `gmail.com` and vice versa — address-level and domain-level ignores unignore at the same level, which is correct.

**Schema bump v2 → v3 with blow-away on mismatch.** `events_state.py` `CURRENT_SCHEMA_VERSION = 2 → 3`; the existing mismatch branch drops the cache and prints a WARNING, no migration code. The alternative — fall back to `sender_block_key = sender_domain` in `classify()` when the field is missing on cached events — would have produced the mixed-granularity UX described above during the 120-day GC window. One run of 60-day re-extraction is the accepted cost; Tom watched the first-run logs ("all 66 candidates to the agent") and confirmed the behavior matches the design-note intent. `test_load_state_v2_on_disk_is_blown_away` pins the policy alongside the pre-existing v1 blow-away test.

**Existing sheet rows not migrated.** Any `gmail.com` / `yahoo.com` / other freemail-domain rows accumulated in the Ignored Senders tab pre-#20 continue blocking the entire domain until Ellen deletes them manually. Named explicitly in the design note, the session notes during the rollout, and the live-verification instructions. Automating the cleanup would have required Apps Script code that reads `freemail_domains.txt` and filters retroactively, which is a one-time operation that's easier to eyeball in the sheet than to write code for.

# 21. Dedupe candidate messages before agent extraction — 9882a1c (design note + [~] flip) / 775f173 (helper + 8 unit tests) / 44283b6 (step2b wiring + integration test)

Filed 2026-04-17 after Tom saw a single dance-studio "Re: First dibs on Recital TICKETS…" thread produce four hits (`[31/66]`, `[32/66]`, `[35/66]`, `[36/66]`) in the extractor input. The five Gmail query templates in `scripts/build_queries.py` overlap by design, and agent cost scales roughly linearly with candidate count, so redundant candidates were real spend on every run.

**Correction to the ROADMAP framing.** The original entry pitched a "messageId-level dedup" fix; discovery showed `step2b_read_promising` already has a `seen_ids: set[str]` pass collapsing messageId duplicates across the five query result sets. The four dance-studio hits were therefore four *distinct messages in the same Gmail thread* — replies to the original with different `messageId`s but the same `threadId`. The real fix was `threadId`-level dedup, the "slightly more ambitious" option named in the same ROADMAP entry.

**Policy: collapse by `threadId`, keep the latest message per thread.** Tom's explicit call: most reminder threads restate the operative date/decision in the latest reply, so discarding earlier replies saves agent cost *and* a `read_message` Gmail-API call per dropped stub. Sort key is the parsed stub `Date` header via `email.utils.parsedate_to_datetime`; ties break on first-seen order; unparseable/missing `Date` loses to any parseable peer; an all-malformed group falls back to first-seen. Stubs with empty/missing `threadId` pass through ungrouped. Dropped stubs are NOT marked as processed in `events_state` — "un-mark when a newer reply becomes the new latest" is brittle, and on the next run Gmail re-returns them and step2b re-drops them cheaply.

**Placement: after the existing messageId pass, before `read_message`.** `_dedupe_by_thread` is pure (unit-testable without mocks), lives in `main.py` alongside the other step helpers, and fires inside `step2b_read_promising` after `seen_ids` and before the body-fetch loop. That placement saves Gmail API calls, not just agent spend. The one pre-existing `Unique messages to read: {N}` log line was replaced by a three-line funnel — stubs collected / unique messageIds / after thread dedup — so the compression ratio is visible in the Actions log, and a regression that made thread dedup a no-op would show identical counts on lines 2 and 3.

**Escape hatch for thin collapsed-thread survivors.** `python main.py --reextract <messageId>` forces a specific messageId back through the agent; the cache-eviction path preserves it past the thread-dedup filter. Used when the digest's outlier-alerts block or a missing page item suggests the latest reply was too thin to carry the operative detail.

**Live verification.** Tom signed off 2026-04-17 after running the first live cron post-merge, eyeballing the three-line funnel in the Actions log, and confirming the rendered page had no missing items that should have survived. The dance-studio thread compressed from four stubs to one.

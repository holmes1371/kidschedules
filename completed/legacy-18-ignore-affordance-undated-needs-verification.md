# 18. Ignore affordance for undated "Needs Verification" cards — 41505aa (markup + tests + ROADMAP flip) / aade8aa (ignored_n count extended to undated)

The "Needs Verification" section at the bottom of the page renders cards for events the extractor surfaced without a date. Those cards went through `_undated_card` in `scripts/process_events.py` and carried no Ignore button — dated cards had one, undated cards did not. Closing this asymmetry was the whole scope: extend the existing per-event ignore mechanics to undated cards, reuse the stable-id scheme, the `ignored` localStorage list, and the Apps Script webhook POST. The "Ignore sender" variant stayed out of scope — undated cards that reach this section are typically the ones the extractor couldn't fully ground, so their sender domain is unreliable.

No design note was written (the ROADMAP entry explicitly waived it: "straight port of the existing pattern"). The flagged risk was subtle id-collision across dated/undated sections; that contract is now pinned by `test_event_id_undated_vs_dated_no_collision` — `_event_id(name, "", child)` and `_event_id(name, "YYYY-MM-DD", child)` are disjoint by construction because the sha1 input differs on the middle segment.

`scripts/process_events.py::_undated_card` (`41505aa`) now mirrors `_event_card`'s top-action-row pattern: `data-event-id` on the outer div, `data-ignored="1" data-ignored-reason="event"` when `is_ignored`, and an `event-actions-top` div wrapping the Ignore/Unignore button. The `.ignore-status` placeholder div was intentionally NOT added — it's `:empty` → `display:none` on dated cards anyway, and omitting it keeps the undated markup minimal. Button attributes `data-event-name` + `data-event-date` use the normalized values from `classify()` so `data-event-date=""` is emitted on undated cards; the JS handler's `getAttribute("data-event-date") || ""` normalizes empty-or-missing to `""` for the POST. The click router and hydration IIFE already selected on `.event-card[data-event-id]` generically, so no JS handler changes were needed — `setIgnored`/`setActive` don't touch the yellow left-rail color, only `display` + `ignored` class + button swap, so undated cards' visual identity survives toggling.

The server-side `ignored_n` count (`aade8aa`) was extended to span display + undated. Deliberate deviation from "straight port" — without it, an undated card ignored on a prior run would reload hidden on the next build with no toggle available to reveal it. Tom signed off this deviation in the plan. Test coverage: `test_render_html_show_ignored_toggle_counts_undated_only` pins the undated-alone case; `test_render_html_show_ignored_toggle_counts_dated_plus_undated` pins the additive case.

Apps Script `_handleIgnore` was audited and tolerates empty `date` without change: it writes `date || ''` verbatim with no validation; `_listIgnoredEvents` echoes that back; `_load_ignored_ids` only extracts `id`. End-to-end tolerant. Sheet rows for undated ignores land with an empty Date column and that's correct.

Full suite after landing: 312 passed (prior 306 + 1 id-collision + 3 undated render + 2 toggle count = +6). Tom confirmed live QA on 2026-04-17: click Ignore on an undated card hides it and raises "Show ignored (1)"; click Show ignored reveals it with an Unignore button; click Unignore restores it and drops the toggle; reload persists state; the Apps Script "Ignored Events" sheet picks up a new row with `date=""`.

# Order cards by start time within each day

> Shipped via **[PR #23](https://github.com/holmes1371/kidschedules/pull/23)**, no standalone tracking issue was filed — the request came directly from Tom in-session. `#23` is the PR number (GitHub shares the issue/PR numbering space, so it resolves unambiguously).

## What was built

Cards within a single day were ordered alphabetically by name, so a 12:45 PM event could sit above a 9:30 AM one on the same day. Weeks and days were already ascending and correct — only the *intraday* order was broken.

`scripts/process_events.py`:

- **`_event_start_time(ev) -> dt.time | None`** — extracts a start clock-time for sorting. Tries `_parse_time_range` (range start), then `_parse_clock_time` (single time), then a tolerant `_CLOCK_RE.search` fallback. Returns `None` for all-day / "Time TBD" / no-clock strings.
- **`_day_sort_key(ev)`** — `(date, all_day_rank, start, name.lower())`. `all_day_rank` is `0` for untimed (sorts first), `1` for timed. `name.lower()` stays the tie-break, which preserves the pre-existing all-day-only name sort.
- **`group_by_week`** sorts with `_day_sort_key` instead of the old `(date, name)` lambda.
- **`_clock_match_to_time(m)`** — extracted the `_CLOCK_RE`-match → `dt.time` conversion so `_parse_clock_time` and the search fallback share it.

Three tests added to `tests/test_process_events.py` (start-time ordering incl. a range, all-day-before-timed, and the trailing-text regression). Full suite green at 896.

## Decisions worth preserving

- **All-day cards lead the day, then timed earliest-first.** Tom's call (Google-Calendar all-day-row-on-top convention) over the alternative of pushing all-day to the bottom. Either kept the existing all-day-only name-sort test green; the rank+name tie-break is what preserves it.
- **Everything ascending.** Tom's initial phrasing said "descending (most recent at top)," but his example (a 12:45 card wrongly above a 9:45) meant the opposite. Clarified up front via AskUserQuestion before writing code — weeks/days already ascending, only intraday was the ask. Worth remembering: the literal words and the example contradicted, and the example was the true intent.
- **Sort tolerance ≠ export tolerance.** The strict `fullmatch` parsers (`_parse_time_range`, `_parse_clock_time`) stay authoritative for `.ics` export — we don't want to invent export times from fuzzy strings. The tolerant `_CLOCK_RE.search` fallback lives *only* in `_event_start_time` (sorting). `.ics` behaviour is unchanged.

## The trailing-text bug (caught live, post-deploy)

First cut used only the strict parsers for the sort key. It passed tests and merged, but on the live page a card reading `10:00 AM – 11:30 AM (approx.)` still floated above a clean `9:45 AM–12:05 PM` card. Cause: the `(approx.)` suffix defeated the strict `fullmatch`, so `_event_start_time` returned `None`, the card landed in the **all-day bucket**, and the all-day bucket sorts *above* timed cards. A clearly-timed card masqueraded as all-day.

Fix: the `_CLOCK_RE.search` fallback grabs a leading clock time from noisier strings (`(approx.)` ranges, `1:30 PM dismissal`). The screenshot Tom sent — with the exact time strings on each card — is what pinned it; the text content of the cards, not just the visual order, was the diagnostic.

Lesson: real-world time strings carry trailing noise the fixture set didn't cover. The first round's tests were all clean strings, so green tests gave false confidence. The regression test now mirrors the screenshot.

## Known limitation (accepted, not fixed)

A range that *both* carries trailing text *and* relies on a shared meridian — e.g. `11-1 PM (approx.)` — sorts by the second time (1 PM) rather than the start (11 AM), because the strict range parser can't fire and the search grabs the first fully-qualified clock token. Rare shape; flagged in the PR rather than over-engineered. If it shows up live, make the range search meridian-aware.

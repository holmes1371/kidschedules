# 19. Deterministic kid attribution from grade / teacher / activity — eb65f8a (design note) / 2ee6a17 (module + unit tests) / ad145ba (wiring + render tests)

Tom reported that the 2026-04-16 live run missed attribution for a "6th grade AAP" card (should go to Everly) and a "Cuppett Performing Arts" card (should go to Isla). The extractor prompt already tells the model to use grade / teacher / activity to pick a kid, but the model followed that inconsistently. Per the standing order, roster-based attribution is deterministic and belongs in Python rather than in the prompt.

Scope: a pure `derive_child_slug(ev, signals)` helper in a new `scripts/roster_match.py`, wired into `_event_card` and `_undated_card`. Five signal tiers (name / teacher / grade / activity / school), with grade matching on both current and rising year (so 7th or "rising 7th grader" → Everly, 4th → Isla), activity matching on parenthetical-extracted aliases (so "B2D" → Everly), and a distinctiveness filter that drops signals shared across kids (so "LAES" stays a no-op until the roster has kids at different schools). The `child` field is not mutated — the event-ID hash depends on it, and the audience-line display stays as context next to the kid pill.

Design note at `design/kid-attribution-derivation.md` (`eb65f8a`) captures the full signal priority, alias extraction, distinctiveness, rendering impact, test fixtures, and the responsibility table. The distinctiveness filter is the quiet load-bearing piece: when both kids share LAES, the school-signal tier goes silent for both kids, so "All LAES students" events stay visible under every filter chip instead of mis-attributing to one kid. The filter is tier-agnostic — any signal string appearing in more than one kid's list is dropped from everyone, so a future shared activity would behave the same way.

`scripts/roster_match.py` (`2ee6a17`) owns the pure helpers: `load_roster`, `advance_grade`, `_grade_matches` (word-boundary regex across ordinal/word/bare-digit forms so "16th" does not match "6th", but "rising 6th grader" and "6th-grade field trip" do), `_activity_aliases` (primary name + parenthetical + first-word when ≥ 6 chars, so "Cuppett Performing Arts Center" picks up bare "Cuppett" but "Born 2 Dance Studio" does not pick up ambiguous "Born"), `_school_aliases` (static `_SCHOOL_ALIASES` table with LAES = Louise Archer Elementary), `build_distinctive_signals`, and `derive_child_slug`. `_FIRST_WORD_ALIAS_MIN_LEN = 6` is the knob that divides those two cases. Tier-priority order is `name → teacher → grade → activity → school`. 46 unit tests exercise every helper, the distinctiveness filter, the tie-break priority (grade beats activity when they fire for different kids), and the no-mutate contract on the event dict.

`scripts/process_events.py` (`ad145ba`) initializes the roster + distinctive signals at module load, then derives `(slug, tier)` per event inside a new `_child_markup()` closure. Rendering rules factored through that closure: slug + `name` tier → pill only; slug + non-`name` tier + child text → pill plus "For: <child>" audience line (routing reason stays visible); slug + empty child → pill only; no slug + child text → audience line only; no slug + empty child → nothing. That preserves the visual contract: direct name matches still look clean; audience-string events get both attribution and context; shared-school events still render with no pill and keep the audience line, visible across every per-kid filter chip. 10 render tests cover the two live misses, both rising-grade paths, activity matching in all three alias forms, teacher-last-name matching, the shared-LAES regression guard, the name-match audience-suppression guard, and the undated-card path.

Full suite after landing: 306 passed (prior 250 + 46 roster_match + 10 render). Tom confirmed visual QA on the live GitHub Pages build on 2026-04-17: "6th grade AAP" attaches to Everly, "Cuppett Performing Arts" attaches to Isla, "All LAES students" renders with no pill under every chip, direct name matches stay pill-only.

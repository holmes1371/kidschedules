# 37. Auto-GC the Ignored + Completed sheets — drop past-dated rows — 3bd0cae (design + flip) / 1f8e8d8 (Tier 1) / 228b082 (Tier 2) / 6b48b67 (close-out SHAs) / 018942e (Date-coercion fix)

Filed 2026-04-27 from Tom as preventive sheet hygiene. Both Apps Script-backed sheets ("Ignored Events" #6/#7, "Completed Events" #32) grew monotonically — once an event's date passed, it GC'd out of `events_state.json` daily but its sheet row stayed forever, shipping in every cron's GET response and the page's localStorage hydration. Verified before any code that the on-page "Show ignored (N)" counter was NOT affected (it's computed from rendered events, not sheet rows), so this was strictly sheet hygiene + GET-payload size. Plan approved 2026-04-28; design note `design/auto-gc-sheets.md`.

**Two-tier shipped together.** Tier 1 alone solves the runtime / payload concern but leaves the sheets growing at the source; Tier 2 keeps the sheets tidy too — one-and-done.

- **Tier 1 (lazy filter at sync time).** New `scripts/sync_ignored_events.py` (promoted from a 25-line inline bash step in `weekly-schedule.yml`, modeled one-for-one on `sync_completed_events.py` / `sync_ignored_senders.py`); both Python sync helpers grew a `_drop_past_dated(rows, today)` that drops rows whose `date` parses as ISO-8601 and is strictly before today. Defensive default: rows with empty / malformed / unparseable `date` pass through (matches `events_state.gc_state`'s posture for undated events). `today` is passed in (no `dt.date.today()` inside the helper) so tests freeze it.
- **Tier 2 (server-side GC).** New `gcPastDatedRows()` in `apps_script.gs` walks "Ignored Events" + "Completed Events" bottom-up and deletes rows whose date column parses and is strictly before midnight-local today. **Does NOT touch "Ignored Senders"** — sender ignores are intentionally permanent. Bound to a daily time-based trigger via the editor (Apps Script editor → ⏰ Triggers → Add Trigger → `gcPastDatedRows` / Time-driven / Day timer / 02:00–03:00). Manual deploy + trigger setup is on Tom; same posture as #34's `?secret=` patch.

**Bug caught in live verification.** Tom's first manual run of `gcPastDatedRows` deleted 0 rows despite many past-dated rows. Cause: when Sheets auto-detects a column as dates, `getValues()` returns native JavaScript `Date` objects, not the YYYY-MM-DD text Apps Script wrote via `appendRow()`. The strict regex parser only accepted strings, so `String(dateObj).trim()` fell into the defensive "unparseable, pass-through" branch on every row. Fix in 018942e: rename `_parseLocalDate` → `_coerceLocalDate`, branch on `value instanceof Date` first (rebuild local-midnight from `getFullYear/Month/Date`), else fall through to the YYYY-MM-DD regex+rollover-guard path for pre-existing string rows. Tom re-pasted, re-ran, confirmed deletions.

**Strict-parse rationale (carried into both tiers).** `Date.parse('YYYY-MM-DD')` returns UTC midnight, which in any timezone west of UTC renders as the *previous* local date — would off-by-one-delete today's rows. Both Tier 1 (`dt.date.fromisoformat`) and Tier 2 (`new Date(y, m-1, d)` with a fields-roundtrip rollover guard) avoid `Date.parse()` / `dateutil`-style coercion to keep the calendar-date comparison local.

**Tests.** Tier 1 only — Apps Script has no unit-test posture in this project. 815 → 872 tests green (57 net new): `tests/test_sync_ignored_events.py` (32 new, full mirror of `test_sync_completed_events.py` plus the `_drop_past_dated` cases), `tests/test_sync_completed_events.py` (~25 added/updated, including the same `_drop_past_dated` surface and a past-dated integration assertion via a `_freeze_today` monkeypatch helper). Two pre-existing tests were updated to use future dates so the new lazy filter doesn't strip them.

**No retroactive bulk purge.** Considered and rejected — daily trigger gets to steady state in a few days; a one-time bulk purge is more destructive (single button, hand-deployed, no testable rollback path) for marginal speed gain.

**Live verification (2026-04-28).** Tom redeployed `apps_script.gs`, set up the daily trigger, and ran `gcPastDatedRows` manually. After the Date-coercion fix landed, the manual run reported a non-zero deletion count and Tom signed off.

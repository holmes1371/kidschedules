# 38. Bug: location linkifier mis-handles email addresses and trailing URL characters — d4581af

Filed 2026-05-01 from Tom — caught on the live page: a card with location `Submit to swimteam@hmsrc.org` rendered the bare-domain segment as `<a href="https://www.hmsrc.org">hmsrc.org</a>`, leaving `swimteam@` as plain text. Same session, Tom flagged a SparkPost tracking URL whose trailing `~~` was outside the anchor — clicking the truncated href landed on a generic fallback page. Two related defects in the same function.

**Bug 1 root cause.** The pre-#38 regex (`_INLINE_URL_RE`) had no email pattern. Its leading `\b` (word boundary) matched the boundary between `@` and `h` in `swimteam@hmsrc.org`, so the URL pattern started mid-address and matched only `hmsrc.org`. The agent emitted the address correctly; the linkifier just couldn't see emails as a thing.

**Bug 2 root cause.** The pre-#38 regex's *trailing* `\b` couldn't anchor after a non-word character like `~`. When a URL's path ended in `~~`, the regex engine backtracked past the trailing tildes to find a position where `\b` could match (between the last *word* char and the first non-word char). The trailing `~~` got stripped from the match, breaking SparkPost's tracking redirect (which requires the exact path). Same class of bug would bite trailing `=` (base64 padding), `&`, `+`, `#`, etc. — all real URL characters.

**Fix.** Replace `_INLINE_URL_RE` with `_EMAIL_OR_URL_RE` — alternation of an email pattern (listed first, so `swimteam@hmsrc.org` is consumed as a single email match before the URL engine can start mid-address) and the URL pattern with no trailing `\b`. Add a `_TRAILING_URL_PUNCT = ".,;:!?)]}"` post-strip on URL matches so `Visit foo.com.` still drops the trailing period into plain text — sentence-punctuation hygiene preserved without the regex idiom that ate real URL characters. Tilde, equals, ampersand, hash, slash, plus all stay in the URL (they're real URL characters in practice — SparkPost uses `~~`, base64 padding uses `==`, query strings use `&`, fragments use `#`). The two `_INLINE_URL_RE.search()` fast-path call sites in `_event_card` and `_undated_card` migrate to `_EMAIL_OR_URL_RE.search()` (semantics identical for the "is there anything to linkify?" check).

**Tests.** 7 new tests in `tests/test_process_events.py` pinning: email→`mailto:` with full address as visible text + href; SparkPost-shape trailing `~~` stays inside href; trailing sentence punctuation falls outside; trailing `==` in query strings stays inside; mixed email + URL in one location renders both as right anchor types; non-overlapping (no double-match producing both `mailto:` AND a separate bare-domain anchor); end-to-end through `render_html` with a real-shape event.

**Live verification (2026-05-01).** Tom triggered a workflow_dispatch rebuild after the push and confirmed: the email address renders as a `mailto:` anchor; the SparkPost URL's trailing `~~` is preserved verbatim in the href on the live page (curl-confirmed against `https://holmes1371.github.io/kidschedules/`). The doctor's-office card still ends up at a broken Google fallback when clicked, but that's because SparkPost itself responds with `Location: https://www.google.com/?!=!` — confirmed via `curl -I` on the URL. Advanced Pediatrics' email vendor configured a tracking URL pointing at a malformed destination; nothing the kids-schedule pipeline can fix on our side. Tom signed off on the linkifier; the broken-destination edge case is out of scope.

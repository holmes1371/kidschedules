# 26. Auto-blocklist must never block parents' personal addresses — c829e2a (design + flip) / 437fa6b (is_protected address-form + 6 tests) / 9a3940c (auto-blocklist unification + 2 integration tests) / cb64dd6 (parent addresses + agent NEVER-flag) / 39c48b6 (session summary)

Filed 2026-04-24 mid-session after item 25's kid_names query landed working but failed live verification because Ellen's emails never reached Gmail's result set. Investigation showed `blocklist_auto.txt` carried `ellen.n.holmes@gmail.com` since 2026-04-14: the agent had flagged a tax-related Ellen-to-Ellen email as adult-only (correct per-email judgment, wrong policy — parents' personal addresses must never be auto-blocked because they're the carrier for self-notes). Two structural problems converged: no per-address protection mechanism existed, and `update_auto_blocklist.py` had a parallel hardcoded `PROTECTED_SUFFIXES` list NOT synchronized with `protected_senders.txt`.

**Fix part 1 (437fa6b): `is_protected` extended for address-form patterns.** A pattern containing `@` is now treated as a full-lowercase-address match (e.g. `ellen.n.holmes@gmail.com`); bare-domain and `*`-suffix patterns unchanged. The matcher only fires the address path when the sender is itself an address — a bare-domain sender does NOT match an address-form pattern, so we don't accidentally protect every mailbox under a freemail domain (load-bearing for the freemail-block mechanism from #20). Bare-domain matching also gained subdomain coverage with a `.`-boundary guard — `fcps.edu` now protects `elementary.fcps.edu` but NOT `notfcps.edu` — preserving the OLD `update_auto_blocklist._is_protected` semantic that was about to be deleted.

**Fix part 2 (9a3940c): unify `update_auto_blocklist` with the shared matcher.** Dropped the parallel `_is_protected` helper and `PROTECTED_SUFFIXES` tuple; gating now flows through `protected_senders.is_protected(addr, patterns)` with patterns loaded from `--protected-senders` (default repo-root `protected_senders.txt`). Single source of truth across all three consumers (`process_events.py`, `build_queries.py`, `update_auto_blocklist.py`). Address-form rejection in `main()` reports `protected sender (...)` in the audit log (vs `protected domain (...)` for bare/suffix matches) so the rejection reason carries the pattern shape forward. `main.py` wires `--protected-senders` through `step3b`. The 7 legacy unit tests for the removed `_is_protected` helper were deleted; same semantics now live in `tests/test_protected_senders.py` (which also gained 1 new subdomain-matching pin). Two new integration tests pin the address-form rejection path and the per-address granularity.

**Fix part 3 (cb64dd6): parent addresses + agent NEVER-flag.** `protected_senders.txt` grew two address-form entries (`ellen.n.holmes@gmail.com`, `thomas.holmes1371@gmail.com`) under a new "Family senders (never auto-block)" comment block; header documentation expanded to describe the address-form pattern shape. `agent.py::_EXTRACTION_BASE_PROMPT`'s NEVER-flag list now names the parents' addresses explicitly with a short rationale — belt-and-suspenders so the suggestion never reaches the gating layer to begin with, keeping the audit log clean.

**State-branch cleanup.** The gating fix prevents future re-adds; the historical Ellen row in `blocklist_auto.txt` on `origin/state` had to be evicted manually. Tom did this 2026-04-24 via the GitHub web UI (one-line removal) — no destructive action required from the agent.
